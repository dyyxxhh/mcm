## Request summary

User wants MCM to grow from a Minecraft mod manager into a broader Minecraft manager while preserving flexible mod-manager/manager use. Requested command surface: `upgrade`, `full-upgrade`, `source add/remove/info`, `pkg dl/download/share/install/make`, top-level `config` interactive config editor, `game install/remove/info/rename/config` plus a default/current game selector (user forgot the English word), `do [file]` to execute `.mcm` files in current directory or a named file, and `run` to launch default game. User has mapped `mc.dyyapp.com` to port `8950`. `pkg share` should provide a company-wide package uploader/downloader for JSON `.mcm` packages, with strict size limit and human verification. User provided Cloudflare Turnstile site/secret-like tokens and says PM2, MySQL, SQLite, Redis are available; MySQL password was provided in chat and must not be written into plan artifacts.

User clarified: frontend/design phase should have the execution agent run `npx getdesign@latest add ollama` and load it; planner must not run it now. For CLI human verification, the CLI should print a verification URL, user opens it, completes Turnstile/browser verification, receives a short code or equivalent, then enters it into CLI to continue. User plans AGPLv3 and open source; compatible code can be considered where license permits, but HMCL/PCL licenses must be respected. Downloads/install must be retry-tolerant and resumable because HMCL/PCL-style downloads may repeatedly fail and should not require repeated manual clicking. Command set may grow beyond the listed commands and should leave room for APT-like subcommands. Share service implementation is delegated to planner best judgment. MCM should be a full Minecraft manager with default install root under the user's home in an `mcm` folder, containing config/saves/etc. with configurable locations. Upload is open and specifically must NOT require an admin token: anyone can upload with IP rate limiting + Turnstile + size limit; anyone can download, with optional bandwidth/rate limiting; package names must be unique. User clarified a speech-recognition error caused earlier confusing wording; the intended requirement is simply “upload does not need admin token.” Web UI should show one-click install commands after upload: `curl -fsSL mc.dyyapp.com/install | bash` installs MCM itself, and package install should use a permanent human-readable package URL like `mc.dyyapp.com/install/pkg/<package-name>` that drives the normal package install flow (install MCM if missing, download version/mods/shaders/resources declared by `.mcm`, and optionally launch game only if the package explicitly requests it and the user confirms/allows it).

User stopped plan generation before it started and added a scope change: MCM must support custom sources so any computer/server can act as a package/mod/version source, not only big providers like CurseForge/Modrinth. Sources are manually imported; even the author's own MCM source is not built in by default. The 8950 server should be one configurable service binary/process that can run share-only, source-only, or both share+source. Source service is optional and lives beside the share service, using the same deployment surface where appropriate. User clarified “Siri” was speech recognition error; intended term is share service + source service. User also clarified source trust model: manually importing a source means the user fully trusts it; design should not be hostile/distrustful after import. However, dangerous actions from a source, such as executing scripts, require a warning and second confirmation by default unless `-y/--yes` is supplied.

## Open owner decisions

- D1 Share service implementation: user delegated choice to planner. Decision: keep it in Rust same repo as the CLI using an Axum-style HTTP server binary/subcommand managed by PM2, because the repo is already Rust, the share service is small, and this avoids a separate Node product stack. Execution may add a tiny frontend/static asset build only if needed for the Turnstile/device-code page.
- D2 Game/install scope: user wants complete Minecraft manager, not just directory manager. Decision: implement full game/version install/run architecture in phases: first generalized model and local instance root under `~/mcm`, then Minecraft version + loader + Java runtime download/install, then launch/auth flow. Do not stop at “existing directories only.”
- D3 Package manifest privacy: shared packages are convenience-first but must not leak secrets. Decision: shared `.mcm` uploads are public/open and exclude account credentials/tokens/secrets by schema. Personal settings/history are not uploaded by default; local/private export can explicitly include selected settings/history when requested.
- D4 Share access policy: user clarified upload is open, not admin-gated, and explicitly does not need an admin token. Decision: anyone can upload if Turnstile passes, IP rate limit passes, body size/schema passes, and package name is unique; anyone can download with optional download throttling/rate limits. No accounts/admin token in first wave.
- D5 Storage backend: planner decision. Decision: SQLite metadata + filesystem JSON package blobs under the server data dir outside `/x`; Redis optional for IP/device-code/rate-limit state if present; no MySQL in first wave unless operational scale later demands it.
- D6 server `/x` boundary: user clarified this only applies to the server side. Server shared package JSON must live outside `/x` because `/x` has daily auto-backup and auto Java install etc. Client-side storage can follow ordinary user-data/config defaults. Recommended default: store uploaded/shared server JSON under an explicit server MCM data directory outside `/x`, e.g. `/var/lib/mcm-share/packages`; local CLI/client may use normal user data dirs.
- D7 Privilege escalation: user clarified root-required operations should ask user to elevate instead of directly failing. Recommended default: detect missing privileges, print the exact command that needs `sudo`, and for interactive CLI offer to re-run via `sudo`/`pkexec`; non-interactive commands fail with actionable escalation instructions rather than silent direct sudo.
- D8 Launcher reference/license: user requested reference to HMCL and PCL launchers with license care. Recommended default: use HMCL/PCL only for conceptual UX/feature comparison, cite licenses, and forbid copying code/assets/text unless license-compatible review explicitly approves.

## Adopted defaults so far

- Rename the old `profile` concept to `game` in user-facing CLI while keeping `profile` as backwards-compatible alias for one release.
- Use English subcommand for default version as `default`: `mcm game default [name]` to show/set selected default game.
- `dl` should be an alias of `download`.
- `full-upgrade` upgrades every configured game/profile; `upgrade` upgrades only the active/default game.
- `.mcm` is JSON, schema-versioned, size-limited, and validated before execution/import.
- `do` with no argument scans current directory for exactly one `*.mcm`; errors if zero or multiple, rather than executing arbitrary multiple files silently.
- Server shared/uploaded `.mcm` JSON is stored outside `/x`; client/local package storage is flexible and can use normal MCM user data directories. `/x` is treated as out-of-scope managed server infrastructure with its own backup/Java automation.

## Resolved decisions

- Root-required actions ask for elevation or provide an exact elevation command; they do not fail generically just because root is needed.
- Execution worker, not planner, must run `npx getdesign@latest add ollama` before frontend design work and load/use the generated design guidance.
- CLI Turnstile flow uses device-code-style verification: CLI requests a challenge/session, prints a URL, user opens browser, completes Turnstile, receives or authorizes a short code, CLI accepts the code or polls until verification succeeds, then proceeds.
- AGPLv3/open-source target permits considering GPLv3-compatible code, but HMCL/PCL reuse still requires explicit license review. Default remains clean-room; direct HMCL code/assets only if GPL/extra-term obligations are intentionally accepted. PCL/PCL2 code/assets remain no-copy due custom restricted license.
- Downloads/install operations must be resumable/retry-tolerant: retries with backoff, partial file validation, resume where supported, staged writes, clear progress, and no repeated manual clicking for transient failures.
- CLI command model must leave room for more APT-like subcommands beyond the initial list; avoid hard-coding the current list as exhaustive.
- Package names in the share service are globally unique in the first-wave namespace; upload of an existing name returns a conflict with suggested alternatives or requires a future explicit version/update mechanism.
- Web UI after upload displays copyable install commands, including one-key MCM bootstrap `curl -fsSL https://mc.dyyapp.com/install | bash` and package install via permanent package-name URL `curl -fsSL https://mc.dyyapp.com/install/pkg/<package-name> | bash` (or equivalent direct command). The route slug is the unique package name, not an opaque temporary short code.
- `/install` script must be safe-by-default and transparent: POSIX shell, `set -euo pipefail` where portable, detects OS/arch, downloads a versioned/signed or checksum-verified MCM release, installs under user-writable location by default, and asks/provides exact sudo command only when a system-wide path requires root.
- `/install/pkg/<package-name>` must execute the normal MCM package flow rather than embedding package logic in shell: ensure/install MCM, then call `mcm do <downloaded .mcm>` or `mcm pkg install <package-name/url>` so schema validation, retries, lock/state, and user confirmations are shared with CLI behavior.
- Package-triggered game launch is allowed only when the `.mcm` explicitly declares it and the CLI either runs interactively with a confirmation prompt or the user supplies an explicit non-interactive flag; no surprise auto-launch from a public URL.
- Permanent install links create security risk, so the plan must include visible trust text on the web page, script preview endpoint or `?dry-run=1`, checksum/signature verification, HTTPS-only URLs, shell injection-safe package-name handling, and tests proving malicious package names cannot affect generated scripts.
- Custom sources are not default-bundled. `mcm source add <url-or-file>` / `source remove` / `source info` manage explicit user-approved sources; a fresh MCM install has no author-owned source preloaded unless the user manually adds it or installs a package that explicitly asks to add a source and the user confirms.
- The 8950 server binary/config supports modes: `share`, `source`, and `all`/`both`. Share mode handles public JSON package uploads/downloads. Source mode serves a read-only source index and artifacts/metadata. Both mode enables both sets of routes from one process and shared storage/config.
- Source service must define a versioned source API and index format separate from `.mcm` share uploads: e.g. source metadata, package/project IDs, versions, file hashes, download URLs, compatibility fields, and optional mirrors. The source index must be cacheable and schema-validated.
- Source trust model: imported sources are trusted by user intent. The client should still parse schemas and verify integrity to catch corruption/bugs, but UX should not treat every third-party source action as malicious. Dangerous operations exposed by a source, especially script execution or root/system changes, require a clear warning and second confirmation by default; `-y/--yes` may skip confirmations for non-interactive use.
