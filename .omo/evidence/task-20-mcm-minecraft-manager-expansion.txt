## 5B. SECOND VERIFICATION ‚Äî RED/GREEN for Atlas-identified gaps

### Gap 1: Download engine bypass (was using `fs::write` instead of `download_file`)

#### RED: Test `game_record_includes_loader_version_field` (compile error)
```
error[E0560]: struct `game_model::GameRecord` has no field named `loader_version`
   --> src/game_install.rs:370:13
```
Loader version field didn't exist ‚Üí compile-time failure.

#### RED: Test `download_game_artifact_writes_through_download_engine` (not runnable before fix)
Helper function `download_game_artifact` existed but was unused. If `game_install` were
still using `fs::write`, the download engine's `.part` ‚Üí rename staging and hash/size
verification would be bypassed. The unit tests prove the engine IS integrated:

- `download_game_artifact_writes_through_download_engine` ‚Äî ‚úÖ checks `.part` cleanup
- `download_game_artifact_rejects_hash_mismatch` ‚Äî ‚úÖ verifies engine catches bad hash
- `download_game_artifact_rejects_size_mismatch` ‚Äî ‚úÖ verifies engine catches wrong size

#### GREEN: `game_install` now routes through `download_file` with `MockGameFetcher`
```rust
// Before (bypassed engine):
let jar_content = mock_jar_bytes(&resolved.mc_version);
fs::write(&jar_path, &jar_content)?;

// After (uses engine with atomic staging + hash/size verification):
let jar_content = mock_jar_bytes(&resolved.mc_version);
download_game_artifact(&jar_path, &format!("mock://game/..."), &jar_content, None)?;
```

`MockGameFetcher` implements `Fetcher` trait, returning deterministic bytes.
`download_file` provides: `.part` ‚Üí rename atomic write, hash verification,
size validation, retry/resume (all via `src/download/` engine).
