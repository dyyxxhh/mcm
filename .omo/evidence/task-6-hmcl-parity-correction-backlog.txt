Task 6: Launch/run compatibility with corrected install layout
==============================================================
Date: 2026-06-29

## Changes Made

### src/launch.rs — render() fix
- render() now includes "-cp <classpath>" between JVM args and main class
- Previously, -cp was filtered from jvm_args but never added to render output
- Classpath is colon-separated, shell-quoted, rendered after JVM args

### src/run_cmd.rs — spawn_game() fix
- spawn_game() now adds "-cp <classpath_string>" between JVM args and main class
- Uses the same colon-separated classpath as render()
- Ensures dry-run and real execution use identical classpath construction

### tests/run.rs — test fixes and additions

1. run_dry_run_offline_mode_uses_stable_uuid_and_zero_token: --sessionType → --userType
   (version JSON template uses --userType, not --sessionType)

2. run_dry_run_online_mode_uses_account_credentials: --sessionType → --userType
   (same reason as above)

3. run_dry_run_with_loader_shows_loader_main_class: updated to verify version JSON
   main class (net.minecraft.client.main.Main) and fabric resolved path in classpath

4. run_without_dry_run_errors_with_not_implemented → run_without_dry_run_spawns_java_and_records_argv:
   Replaced stale "not implemented" test with real spawn verification using fake Java
   that records argv. Verifies -Djava.library.path, main class, --username, and -cp.

5. Added run_uses_complete_installed_layout:
   Comprehensive test verifying:
   - Java executable in dry-run output
   - JVM args (-Djava.library.path, launcher brand/version)
   - Natives dir in path
   - -cp flag with classpath
   - Main class
   - Game args (--username, --uuid, --accessToken, --userType, --gameDir, --assetsDir)
   - Non-dry-run: fake Java records argv including -cp, verifies cwd = game root_dir
