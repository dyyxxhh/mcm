SECTION 6: IDENTIFIED GAPS
===============================================================================

GAP-1: No direct web install button (low priority)
  Description: The Web UI provides the install command (curl | bash) as a copyable
               snippet but does NOT have a "one-click install" button that directly
               downloads and applies the package. The CLI has "pkg install --server"
               which downloads and applies in one step. The Web UI requires the user
               to manually copy and run the curl command.
  Severity: LOW (by design — web install is curl-pipe-bash, not in-browser)
  Evidence: Web UI shows install command in detail page and package list.
  Recommendation: This is intentional. The install command is the web install route.

GAP-2: No package detail in CLI (low priority)
  Description: The CLI has no "pkg detail" command to show package metadata + install
               command. The Web UI has /detail/{slug} with full metadata view, install
               command, and owner management actions.
  Severity: LOW (CLI is command-oriented, not browsing-oriented)
  Recommendation: Not needed — CLI has download and list commands for metadata.

GAP-3: CLI pkg info reads local .mcm only (informational)
  Description: `pkg info <path>` only reads local .mcm files. There's no way to get
               info about a remote package by slug from the CLI. The Web UI shows
               package metadata via the detail page.
  Severity: LOW (informational gap, not functional)
  Recommendation: Could add `pkg info --server <url> <slug>` for remote info.

GAP-4: Daily push limit only enforced server-side (informational)
  Description: The daily push limit (1 publish/update per day per user) is enforced
               by the server. The CLI does not check this locally — it sends the
               request and reports the server error. The Web UI handles the 429
               response with a user-friendly message.
  Severity: NONE (correct design — server is authority)
  Evidence: CLI output: "Error: update failed: daily push limit reached"
            Web UI: handles 429 with "Daily publish limit reached" banner

GAP-5: pkg install --server requires active profile (informational)
  Description: `pkg install <slug> --server <url>` downloads the package from the
               server and applies it to the current game profile. If no profile is
               configured, it fails with "no active profile". This is expected
               behavior — the Web UI's install route (curl | bash) handles this
               differently via the install script.
  Severity: NONE (correct design — local state requirement)
  Evidence: "Error: no active profile; run profile add or profile use"
