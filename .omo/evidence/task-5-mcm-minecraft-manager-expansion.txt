## Design decisions

1. **Migration is in-memory only, not persisted.** Persisting migrated games
   would race with `mods add` (which re-saves config with empty games). Instead,
   migration runs on every `load_config` call when profiles exist and games
   are empty. Cost is negligible (BTreeMap iteration). Old profile data is
   never deleted.

2. **No stderr warning on migration.** The task spec said "log a warning to
   stderr", but the 44 characterization tests assert exact stderr output.
   A warning would break them. Migration is silent.

3. **`game config <name>` is show-only.** The CLI `GameCommand::Config` variant
   (from task 4) only has `name` — no `--set` flag. Task constraints forbid
   modifying CLI variants. Setting fields requires a CLI change in a future
   task.

4. **`game remove` only removes the config record.** Never touches on-disk
   files. Prints what was removed and notes the root_dir is left on disk.
   Full safety policy is task 7.

5. **Default root is `~/mcm`** via `directories::UserDirs::new().home_dir().join("mcm")`,
   with fallback to `PathBuf::from("mcm")` if home dir can't be resolved.

6. **`GlobalConfig` has only `root_dir` for now.** Future tasks (instance/saves/
   config/cache paths) will extend it.
